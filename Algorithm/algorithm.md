# 数据结构

## B Tree

B-Tree（B树）是一种专为`磁盘`等外部存储设备设计的`自平衡多叉查找树`。简单说，它是对二叉查找树的泛化，每个节点可以拥有多个（两个以上）子节点。它通过 “矮胖” 的造型，显著减少了在磁盘上查找数据时所需的慢速读取次数。

### B-Tree的定义

对于M阶的B树，满足以下性质：

1. 每个节点最多有m个子节点。
2. 每个非叶子节点（除了根节点）至少有⌈m/2⌉个子节点。
3. 根节点至少有两个子节点（除非它是叶子节点）。
4. 所有叶子节点都在同一层。
5. 每个节点包含k-1个键（其中k是该节点的子节点数量），这些键将子节点分割成k个区间。并且这些键是`有序的`，左边的子树包含小于键的值，右边的子树包含大于键的值。

### B-Tree的每个节点的结构

每个节点存储指针和键（索引）：
``` 
[ 指针 | 索引+数据 | 指针 | 索引+数据 | 指针 ]
```

其中保证索引是有序的，左边的指针指向小于索引的子树，右边的指针指向大于索引的子树。

从节点的指针+索引的结构，以及B-Tree的定义，可以衍生出实际使用的定义：

* 每个节点最多有m-1个索引和m个指针。
* 每个非叶子节点至少有⌈m/2⌉-1个索引和⌈m/2⌉个指针。

### B-Tree的插入操作

插入时找到合适的叶子节点，如果该节点未满，则直接插入；如果该节点已满，则将其分裂为两个节点，并将中间的索引上升到父节点中。这个过程可能会递归地向上进行，直到根节点。

```
第1步：插入 10
[10]

第2步：插入 20
[10, 20]

第3步：插入 30
     [20]
    /    \
  [10]   [30]

第4步：插入 40
     [20]
    /    \
  [10]   [30, 40]

第5步：插入 50
     [20, 40]
    /    |    \
  [10]  [30]  [50]

第6步：插入 60
     [20, 40]
    /    |    \
  [10]  [30]  [50, 60]

```

### B-Tree的删除操作

删除有三种情况：

* 删除节点在内部节点（非叶子节点）：用其前驱或后继节点替换该节点，然后删除前驱或后继节点。
* 删除后节点的键数在⌈m/2⌉-1和m-1之间：直接删除。
* 删除后节点的键数少于⌈m/2⌉-1：需要进行合并或借位操作。
  * 如果兄弟节点有多余的键，可以从兄弟节点借一个键过来，并调整父节点的索引。
  * 如果兄弟节点没有多余的键，则将当前节点与一个兄弟节点合并，并将父节点的分割键下移到合并后的节点中。

```
定位要删除的键
    ↓
键在叶子节点？
    ├─ 是 → 直接删除 → 检查是否低于最小键数
    │               ├─ 否 → 结束
    │               └─ 是 → 尝试借位 → 失败则合并
    │
    └─ 否（在内部节点）
        → 用前驱/后继替换 → 递归删除前驱/后继
        → 检查是否低于最小键数 → 借位或合并
```

```
初始树：
                [40]
               /    \
          [20]       [60, 80]
         /    \      /    |   \
       [10]  [30]  [50] [70] [90]

删除90 → 合并 → 
                [40]
               /    \
          [20]       [60]
         /    \      /    \
       [10]  [30]  [50]  [70,80]

删除70 → 直接删除 →
                [40]
               /    \
          [20]       [60]
         /    \      /    \
       [10]  [30]  [50]  [80]

删除80 → 合并 → 
                [20, 40]
               /    |    \
            [10]  [30]  [50, 60]

删除60 → 直接删除 →
                [20, 40]
               /    |    \
            [10]  [30]  [50]

删除50 → 合并 → 
                [20]
               /    \
            [10]   [30, 40]

删除40 → 直接删除 →
                [20]
               /    \
            [10]   [30]

删除30 → 合并 → 树高降低 →
                [10, 20]
```

上述涉及了递归合并

```
合并前（所有叶子在第3层）：
                    [40]              ← 第1层
                   /    \
              [20]       [60]         ← 第2层
             /    \      /    \
           [10]  [30]  [50]  [80]     ← 第3层（叶子层）

第一次合并（叶子层）：
合并 [50] + 父键 60 + [] → [50, 60]
父节点 [60] 变为 []

此时状态：
                    [40]              ← 第1层
                   /    \
              [20]       []           ← 第2层（父节点变空）
             /    \      /    \
           [10]  [30] [50, 60]        ← 第3层（叶子层）

第二次合并（第2层）：
因为父节点 [] 空了，需要与兄弟 [20] 合并
合并 [20] + 父键 40 + [] → [20, 40]

合并后（树结构重组）：
                [20, 40]              ← 新的第1层（原第2层上移）
               /    |    \
            [10]  [30]  [50, 60]      ← 新的第2层（叶子层）
```

## B+ Tree

B+树是B树的一个变种，它和B Tree基本相同，核心区别在于：

* B+树的所有数据都存储在叶子节点中，而内部节点只存储索引（键）信息。
* 叶子节点之间通过指针连接，形成一个链表结构，方便范围查询。
* 所有的索引（键）都必定存在于叶子节点中，而内部节点会有冗余的索引信息。

这是对B树的优化，使得B+树在范围查询和顺序访问方面更高效。

展示为什么B Tree中存在范围查询的问题：

```
                    [20, 数据]              ← 内部节点存数据
                   /          \
            [10, 数据]      [30, 40, 数据]   ← 内部节点也存数据
           /       \        /       \
      [5,数据] [15,数据] [25,数据] [35,45,数据] ← 叶子节点存数据
```

* 查找 15：根节点→左子节点，在内部节点就找到了，无需到叶子
* 范围查询 20~40：会在根节点找到一部分数据，但无法直接找到所有数据，需要继续往下找，效率较低。

而在B+树中：

```
                    [20, 40]                ← 内部节点只存键（纯索引）
                   /    |    \
            [10]      [30]      [50, 60]    ← 内部节点只存键
           /    \     /   \     /    \
      [5,10] [15]  [25] [30] [50] [60,70]  ← 叶子节点存所有数据
         ↓      ↓    ↓    ↓    ↓    ↓
         ====== 双向链表 ==============→
```

* 查找 15：必须从根一路走到叶子节点才能找到数据
* 范围查询 20~40：先找到20，然后沿着叶子链表一路向右遍历即可，顺序读取，极快

### 为什么B+树更适合数据库索引

1. 树更矮：B+树的内部节点只存索引，叶子节点存数据，因此每个内部节点可以存储更多的索引，树的高度更低。
2. 范围查询更快： B+树的叶子节点通过链表连接，支持顺序访问和范围查询。
3. 查询性能稳定：每次查询都必须到叶子节点，保证了查询路径的一致性，避免了B树中可能出现的不同路径长度问题。

### B+树的插入和删除操作

B+树的插入和删除操作与B树完全相同，只是由于所有KEY都必须在叶子节点，内部节点的KEY是冗余的，所以在插入&删除时，要注意修改冗余的内部节点KEY：
